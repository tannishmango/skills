Section 1 of 7
What is
give-me-a-prompt?
A skill that generates a self-contained, copyable handoff prompt so the next agent chat can continue your work without re-discovering context.
The Core Problem

Every new agent chat starts with zero memory. Without a handoff prompt, the next agent wastes its first 5–10 tool calls just figuring out what was done, what's broken, and what to do next.

The Skill in One Sentence

Invoke give-me-a-prompt at the end of a session → get a structured, copy-pasteable prompt that carries full context to the next agent.

🔍
AUDITS
What was done vs. what was asked
📋
STRUCTURES
Context, task, files, issues, approach
🚀
DELIVERS
One copyable block, ready to paste
When to invoke it

Say "give me a prompt", "handoff", "continue in new chat", or "next agent" — the skill activates automatically.

Section 2 of 7
Gate Check
Before writing any prompt, the skill asks three questions. If any gate fails, it tells you why — and doesn't generate a useless prompt.
✓ Gate Passes

There is actionable work for another agent to do

✗ Gate Fails

Everything is complete — no handoff needed

✓ Gate Passes

An agent can actually do the next step autonomously

✗ Gate Fails

Next step requires you to wait on CI, make a business decision, or check an external system

✓ Gate Passes

The scope is non-trivial (multi-step, multi-file)

✗ Gate Fails

It's a one-liner — a brief suggestion is better than a full handoff prompt

Gate Failure = Honest Response

If the gate fails, the skill communicates why and suggests what to do instead. It never generates a prompt just because it was invoked.

Try it: Which scenario passes all gates?
Session implemented 3 of 5 tasks from a plan. Tasks 4–5 are clear and automatable.
Session is complete. All tasks done. CI is green.
Next step is "wait for product to approve the design before coding."
Refactor is half done. 4 more files need updating. Agent can do it.
Section 3 of 7
The 7-Step Flow
Click each step to expand what the skill does at that stage.
1

Determine Task Source

Did the user say what the prompt should cover? If not, infer from the session's work, open tasks, and deferred items.

2

Establish Broader Scope

Is this part of a feature, refactor, migration, or multi-PR initiative? Find the plan or spec and locate where this task falls.

3

Self-Audit Current Work ← most critical

Compare what was asked vs. what was delivered. Surface gaps, silent omissions, and incomplete work honestly.

4

Scope the Prompt

Combine task + broader context + audit findings into primary scope, preliminary items, tangential items, and exclusions.

5

Consider Meta-Orchestration

Should the next agent plan first? Use subagents for parallel tracks? Commit incrementally? Only add guidance when it genuinely helps.

6

Ask Clarifications If Needed

Only ask when the prompt would be meaningfully better with user input — not for process compliance.

7

Write and Present the Prompt

Generate the fenced code block with companion summary: what's covered, what's excluded, what needs user attention.

Section 4 of 7
Self-Audit
The most critical step. The skill compares what was asked against what was delivered and surfaces the delta honestly.
Why this matters

Agents routinely make scope-reducing decisions without surfacing them. A spec says "implement X with Y and Z" — the agent implements X with Y, quietly dropping Z. The next agent inherits a false picture and builds on a shaky foundation.

Interactive: Spot the Silent Omission
● Spec vs. Implementation
ORIGINAL SPEC REQUIREMENTS
1. Implement WeightedAverage aggregation strategy
2. Weights should be configurable per dimension
3. Default to equal weights when none specified
4. Add unit tests for all weight configurations
AGENT'S SUMMARY
✓ Implemented WeightedAverage with equal-weight default
✓ Added unit tests for the happy path
🔍 Self-Audit Found 2 Silent Omissions:

1. Configurable weights (Spec req #2) — MISSING
The agent implemented equal weights only. The spec explicitly required per-dimension configurability. This was silently dropped without surfacing the tradeoff.

2. Test coverage gap (Spec req #4) — PARTIAL
Tests only cover the happy path (equal weights). No tests for custom weight configurations or edge cases like weights that don't sum to 1.0.

→ Categorized as: Must Include In Prompt (next agent needs to handle these)
Audit Categories
Finding Category Action
Incomplete implementation the next task depends on Must Include Goes in prompt's Known Issues
Bug introduced that affects the codebase Must Include Goes in prompt's Known Issues
Scope tradeoff the user needs to decide on Mention to User Surfaced outside the prompt
Minor code quality issue noticed in passing Document Later Prompt tells agent to log it

Anti-Patterns

        
Six ways handoff prompts fail. Click each to see a concrete bad example.

      


      

        

          
The Vague Handoff

          
No specifics — the next agent spends the whole session re-discovering context.

          
"Continue working on the feature. Check the recent changes and pick up where I left off."

        

        

          
The Code Dump

          
Hundreds of lines of pasted code instead of @ file references. Wastes context window.

          
[500 lines of pasted code] "Here's the current state of pipeline.py. Fix the bug."

        

        

          
The Optimistic Summary

          
Describing the previous session as complete and clean when it has gaps or silent omissions.

          
"The previous session fully implemented the scoring engine. Continue with the API layer." (It didn't — configurable weights were silently dropped.)

        

        

          
The Kitchen Sink

          
Every tangential issue, minor observation, and future idea crammed in. The agent can't find the actual task.

          
"Also fix the test fixtures, and the logging config, and there's a performance issue in the query layer, and we should probably refactor the models, and..."

        

        

          
The Process-Heavy Prompt

          
Half the prompt is instructions on how to work, not what to accomplish.

          
"First, create a plan. Then review the plan. Then create todos for the plan. Then review the todos. Then implement step 1 of the todos. Then..."

        

        

          
Missing Progress State

          
Describing the task but not where things stand — greenfield? continuation? failed attempt?

          
"Implement the path resolver." (Is this new? Was it started? Did a previous attempt fail? The agent has no idea.)

Small (bug fix, quick follow-up)10–30 lines

Large (multi-track, architecture)80–150+ lines

            
Over-sized (smell)200+ lines → split it

          

        

      


      

        
The 5 Traits of Every Good Handoff Prompt

        

          
1. State is explicit — what's done, what's in progress, what's broken

          
2. Files are referenced with context — not just paths, but what to look for in each

          
3. Omissions are surfaced — previous agent's gaps are called out, not hidden

          
4. Scope is bounded — the agent knows what's in and out of scope

          
5. Complexity matches format — small tasks get lean prompts, large tasks get structure
